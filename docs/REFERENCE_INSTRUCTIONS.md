# FINAL PROJECT — Instructions & Rubric
**Course:** Software Component Design (SCD) | **Section:** BSSE 2  
**Term:** 2nd Semester, A.Y. 2025–2026

---

## I. Overview
For your Final Project in Software Component Design, your group will design and build a functional system — either a Web Application or a Mobile Application — that meaningfully integrates at least 2–3 of the design patterns discussed throughout the course.

This project is your opportunity to demonstrate that you understand not just the syntax of design patterns, but the **WHY** behind them — knowing when a pattern is appropriate, how it improves your architecture, and how to explain it clearly to others.

**You are not just coding — you are architecting.** Your system must show real, justified use of design patterns. Bolting a pattern onto code where it does not belong will be penalized.

---

## II. Learning Objectives
By the end of this project, students should be able to:
1. Apply at least 2–3 design patterns (Creational, Structural, or Behavioral) in a real working system.
2. Justify each pattern choice — explain why that pattern fits, not just what it does.
3. Collaborate as a team with clear, documented roles and responsibilities.
4. Produce a well-structured documentation paper covering all required sections.
5. Present their system and defend architectural decisions as a group.

---

## III. What to Build
Your group will choose ONE of the following platforms:

| Platform | Description |
| :--- | :--- |
| **Web Application** | Built with any frontend/backend stack (React, Vue, Django, Laravel, Node.js, etc.) |
| **Mobile Application** | Built using Flutter, React Native, Android (Kotlin/Java), or iOS (Swift) |

**Programming Language — Your Choice**  
You may use any programming language your group is comfortable with. What matters is that your design pattern implementation is correct, justifiable, and clearly visible in your code.

**Domain**  
Your system can be about any domain — productivity tools, school systems, e-commerce, healthcare, games, etc. — as long as it is original and non-trivial.

**Progressive Prototype**  
Your system must be a working prototype. It does not need to be fully polished, but the core features and all integrated design patterns must be functional and demonstrable during your presentation.

---

## IV. Design Pattern Requirements
Your system must implement at least **2–3 design patterns** from those covered in SCD.
- **Minimum:** 2 patterns
- **Recommended:** 3 patterns (Bonus consideration for more if correctly applied).

### What Counts as a Valid Pattern Use?
- The pattern must solve a **real design problem** in your system.
- You must be able to explain: What problem existed before? How does the pattern solve it? What design principle does it uphold?
- The pattern's key classes (Interface, Concrete Classes, Context, Subject/Observer, etc.) must be clearly identifiable in your code.

---

## V. Documentation Paper Requirements

| Section | Content Description | What to Include |
| :--- | :--- | :--- |
| **Title Page** | Name of the system | Project title, group members with roles, course, section, date of submission |
| **Member Roles** | Members and responsibilities | A table: each member's name and their specific responsibilities |
| **Problem Statement** | Definition of the problem | Clearly explain the problem being solved, its importance, current challenges, and target users |
| **Application Overview** | System description | A brief but clear description of what the system is and how it generally works |
| **Proposed Solution** | System approach | Explain how the system addresses the problem, including workflow and how design patterns help |
| **Goals and Objectives** | Purpose and targets | List at least 4 objectives |
| **Design Patterns Used** | Application of patterns | Detailed technical breakdown (see below) |
| **System Architecture Diagram** | System structure | A diagram showing how major components are connected |
| **Features** | Functionalities | List major features (CRUD, Auth, etc.) |
| **Project Timeline** | Project planning | A Gantt chart showing phases, duration, and assigned members |
| **References** | Sources | Citations for external code, libraries, and references |

### Design Patterns Used (Detailed Requirements)
For each pattern used, include:
- **Pattern Name & Category** (Creational, Structural, Behavioral)
- **Location in Code** (File names, Class/module names)
- **Problem Without the Pattern** (Coupling, duplication, flexibility issues)
- **Proposed Solution** (How the pattern is applied)
- **How the Pattern Solves It** (Roles of Interface, Concrete Classes, Context, etc.)
- **Design Principle Upheld** (SRP, OCP, DIP, etc. with justification)
- **Code Snippet** (Optional but recommended for key structures)

## VIII. Academic Integrity
- **Originality:** Core logic and design patterns must be written/understood by the group.

- **Citations:** Properly cite all external code and libraries.
